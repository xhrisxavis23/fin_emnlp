# CSI 500 FaVOR — paper Table 1 reproduction report

**Goal**: verify whether the modularized `revision/favor` produces results "meaningfully similar" to paper Table 1, accounting for LLM stochasticity (same `concept`, `outer-loop=1`, three independent runs).

**Setup**
- concept (verbatim from `01_15_new_qlib/runs/20260207_051736/run_config.json`):
  > After a breakout to a new high, a pullback toward the 20-day moving average often serves as support, increasing the probability of price revisiting the breakout level or exceeding it.
- pipeline: `revision/revision/favor/run_pipeline_parallel_per_combo_parallel.py`
- config: `--outer-loop 1`, `n_trials=20`, `combo_workers=4`, `optuna_jobs=1`, `max_combinations_to_evaluate=-1` (no combo cap), `provider_uri=~/.qlib_full/qlib_data/cn_data` (full 2005–2026 coverage from `.qlib.zip`)
- LLM: `gpt-4o`, `temperature=0.7` (default; not seeded)

## 1. Architecture clarification

`concept` (text) is the input. `step0_hypothesis_setting` (`run/pipeline/stage1.py:87`) calls `HypothesisAgent.purpose_hypothesis(concept, hypothesis_memory)`:

- **outer_iter_1**: `hypothesis_memory=None`, `feedback=""` → LLM creates the **first** structured hypothesis (e.g. `BH_Continuation_Breakout_5D_v1`) from the concept.
- **outer_iter_2+**: previous hypotheses + Stage4 feedback are passed in, the LLM proposes a refined alternative.
- Each outer iter runs the full Stage 1→2→3→4 chain.

`--outer-loop 1` means *one shot at hypothesis generation*. If Stage 2 (formula coverage) or Stage 3 (per-ticker monotonicity) fails, the run terminates with **zero combos**. There is no built-in hook in `revision/favor` for injecting a pre-computed `hypothesis.json` (the original Feb-7 hypothesis cannot be reused as-is).

## 2. Combo, Optuna, hyperparameters (recap)

- **combo** = a 5-tuple of formulas, one per observation, generated by `itertools.product` across observation-grouped formulas (`agent/hypothesis_validation_agent.py:117-186`).
- **Optuna**: per-combo study with categorical search on `[0.55, 0.6, …, 0.95]` per formula, `n_trials=50` (paper) / `n_trials=20` (this experiment), maximizing **IS** excess `information_ratio`. OOS is never seen by Optuna (`run/pipeline/stage4.py:1591-1786`).
- **3 evaluation modes**: `optuna` (per-formula tuned), `fixed_q80`, `fixed_q90` (uniform threshold). Paper Table 1 uses `optuna`.
- **Stage 4 backtest** (`qlib_legacy`): per-formula quantile threshold on training panel → AND of all formula conditions → 5-day horizon, stop_loss=−0.10 (paper) / −0.05 (revision default), equal-weight portfolio, qlib `risk_analysis` for IR/AR/MDD.

### 3.3 The combo-selection issue (data snooping)

`01_15_new_qlib/analysis/0203 copy 3.ipynb` ranks combos by `outsample_excess_return_with_cost.information_ratio` (cells 2–10):

```python
ex_flat = pd.json_normalize(df_raw["outsample_excess_return_with_cost"])
...
df.sort_values(by='information_ratio', ascending=False).head(20)
```

That is: **OOS information_ratio is used to pick the table-1 combo**, not IS or validation IR. Confirming this against `aggregated_stage4_results.csv`:

- Paper combo_51 → OOS IR = +0.6470 (rank #1 of 84 combos)
- Paper combo_15 → IS IR = +0.6774 (the actual IS-best) → OOS IR = −0.5621 (catastrophic)

So the paper's reported number is **OOS-oracle**, not what an honest IS-only selection would produce. Reproducing this on a fresh LLM run cannot match unless a similarly upper-tail OOS combo happens to be generated — which our 3 runs show does not happen reliably.

If we instead compare OOS-oracle across the three reproductions:

| | OOS-oracle combo | IS IR | **OOS IR** | OOS AR |
|---|---|---|---|---|
| Paper run | combo 51 | +0.4388 | **+0.6470** | +0.1397 |
| Run 1 | combo 5  | +0.2378 | **+0.2451** | +0.0754 |
| Run 3 | combo 37 | +0.2784 | **+0.4206** | +0.1297 |

Even with OOS-oracle selection (which the paper uses), reproductions yield 38 % – 65 % of the paper's headline IR. With honest IS-only selection, all reproductions are deeply negative.

## 5. Answers to the user's questions

**Does revision/favor work end-to-end?** Yes. With CN data restored (zip extracted to `~/.qlib_full/`) and a small additive change to `run/config.py` to read `FAVOR_QLIB_PROVIDER_URI_CN` from env, run 1 and run 3 produced full Stage 1–4 specs.

**Are LLM-stochasticity-driven results "meaningfully similar" to paper Table 1?** **No.**

1. The paper's hypothesis (`BH_Continuation_Breakout_5D_v1`, 5-formula combos, 84 combos) is one specific decomposition the LLM did not regenerate in any of 3 fresh runs.
2. Even when the LLM produced a structurally similar hypothesis (run 3, 5-formula combos, 66 combos), the IS-best combo's OOS IR was −0.45 (vs paper's +0.65). The OOS distribution medians are around −0.2 to −0.4 across runs.
3. The paper's +0.65 OOS IR comes from picking the OOS-best combo — i.e. the analysis pipeline uses OOS information for combo selection (`analysis/0203 copy 3.ipynb`).

**Recommended actions for EMNLP revision**

1. Replace OOS-based combo selection in `analysis/0203 copy 3.ipynb` with a held-out **validation** subset (e.g. last 6 mo of IS, or 2020-only). `revision/favor`'s log already prints `Train (2015-01-01 ~ 2019-12-31) → Val/Optuna (2020-01-01 ~ 2020-12-31) → Test (2021-01-01 ~ 2025-12-31)` — but Optuna only uses Val for thresholds. The same Val window should also be used to pick the best *combo*.
2. Report Table 1 numbers as **mean ± std over N independent LLM runs**, not a single cherry-picked run/combo.
3. Use `--outer-loop 5` (the original config) for stability — 1-of-3 runs failed at Stage 2/3 with `--outer-loop 1`. The original paper happened to converge on outer_iter_1 (`outer_loop_used: false`), but that's a lucky single draw, not robustness.
4. Consider adding a hypothesis-injection hook to `revision/favor` (env var: `FAVOR_HYPOTHESIS_JSON=path/to/hypothesis.json`) so frozen, named hypotheses can be re-evaluated deterministically.
