## 원인 진단

Pipeline 의 어떤 단계에서도 MDD 를 selection criterion 으로 사용하지 않음.

| stage | 평가 기준 | MDD 사용? |
|---|---|:---:|
| Stage 1 | formula 생성 (statistical pattern) | ✗ |
| Stage 2 | distribution monotonicity (MAG/DIR/VOL/POS) | ✗ |
| Stage 3 | precision, mean_return, sharpe_ratio, s2_ratio | **✗** |

## 1.2 Stage 4: Optuna objective 에 MDD term 추가

### 현재 코드

`run/pipeline/stage4.py:1784`:

```python
return excess_ir   # IS excess IR, vs benchmark
```

→ Optuna 가 IR 만 maximize. threshold 탐색 결과가 IR 에 over-fit, MDD 통제 0.

### 변경안 (3 가지 옵션)

**A. Multi-objective Optuna** (가장 정확)

```python
study = optuna.create_study(directions=["maximize", "maximize"])
# objective 끝에:
return excess_ir, -abs(max_drawdown)
```

- Pareto front 탐색 후 다양한 IR-MDD trade-off 후보 추출
- 최종 selection 은 별도 rule (예: Pareto 위에서 Calmar 최고 선택)
- 비용: trial 당 wall 동일, best 후보 여러 개라 분석 코드 추가

**B. Penalty term** (가장 간단)

```python
lam = 1.0  # tuning needed
return excess_ir - lam * abs(max_drawdown)
```

- λ 가 새 hyperparameter
- λ → 0 면 IR-only (기존), λ → ∞ 면 MDD-only
- ablation: λ ∈ {0.0, 0.5, 1.0, 2.0, 5.0}

**C. Calmar ratio** (가장 깔끔, paper-friendly)

```python
return annualized_return / max(abs(max_drawdown), 1e-6)
```

- 단일 objective, 추가 hyperparameter 없음
- 정의상 "MDD 1 단위당 AR" → publishing narrative 와 자연스럽게 맞음
- 다만 MDD 가 매우 작을 때 분모 explosion 위험 (cap 필요)

### 추천

옵션 **C (Calmar)** 가 단일 metric 으로 깔끔하고 EMNLP narrative 만들기 쉬움 ("paper 가 IR 만 max → 우리는 risk-adjusted Calmar max → MDD 도 안전").

옵션 **A (Multi-objective)** 가 정보량 가장 많음. 단 분석 복잡도 ↑.

### 기대 효과

- pipeline 전체에서 MDD 가 selection criterion 으로 작동하는 **첫 번째 변경**
- Phase 0 의 4 BEATS PAPER 같은 setting 이 다시 등장해도 MDD 깊은 combo 가 Optuna 단계에서 reject 됨
- honest IS-best 선택 시 MDD 도 자연스럽게 통제

### 영향 추정

- **highest impact lever**. 코드 1~3 줄 수정으로 가장 큰 OOS metric 변화 예상
- 검증: best 4 setting (M09_paper_h10, B01_stoploss_005 등) × n_trials=20 × Calmar objective × S1 sweep 4 jobs ~ 1 시간

## 2.4 Stage 1 LLM stochasticity — variance vs mean shift 분리

### 관찰된 사실

같은 setting (B01_stoploss_005 × paper concept × S5 × mini × n_trials=50) 의 단일-seed 재실행:

| Phase | outer_loop | honest OOS IR |
|---|---:|---:|
| Phase 0 (5/10) | 1 | **+1.06** |
| Phase 2 (5/11) | 3 | **−0.19** |

→ 같은 hypothesis_id (`BH_BreakoutPullbackSupport_5D_v1`) 생성됐지만 sub-formula 의 미세 차이로 OOS 가 +1.06 → −0.19 폭변.

### 질문

이게 **variance** (mean 은 0 근처, 단일-seed 가 우연히 +1.06 또는 −0.19 sampling) 인가, **mean shift** (다른 hypothesis path 가 mean 자체 다른가) 인가?

### 제안 실험

- N=10 seeds × 같은 setting (B01 × paper × S5 × mini × ol=1 × n_trials=50)
- mean ± std 계산
- 만약 mean ≈ 0, std 크면 → variance only, paper §4.4 의 10-seed averaging 정당화
- 만약 mean ≠ 0, std 작으면 → mean shift, hypothesis 자체가 path-dependent

### 추가: temperature ablation

- 같은 setting × temperature ∈ {0.0, 0.3, 0.7, 0.9} (현재 0.7~0.9 mix)
- 각 temperature 의 mean ± std
- temperature 가 mean 을 옮기는지 variance 만 줄이는지 정량

### 논문 활용

- paper §4.4 의 "10-seed averaging" 표준화 정당화
- single-seed 결과 (paper Table 1 포함) 의 신뢰도 평가
- LLM agentic factor mining 의 reproducibility 한계 정량화

5. **Temperature 는 variance lever, mean lever 아님** — E[IR|temp] 거의 평탄, Var[IR|temp] 만 차이. ablation 가치는 multi-seed std 통제이지 mean shift 가 아님. `향후_과제.md` 의 2순위 lever 분류와 정합.
6. **LLM training cutoff leakage** — Stage 1 hypothesis 생성이 LLM 학습 데이터 prior 에 의존 가능. paper-aligned 5y/1y/5y (test 2021-2025) 는 모든 modern LLM cutoff 침범. paper concept 5종이 broad 라 직접 영향은 작을 수 있으나 reviewer 의 자연스러운 의심거리.

## 3.1 코드 수정 — 새로 발견된 lever

### 3.1.1 (新추가) `reasoning_effort` 인자 도입 (가벼운 패치, 1시간)

#### 동기

gpt-5.4-mini 같은 OpenAI hidden-reasoning 모델에서 temperature 가 silent no-op 인 상황의 진짜 lever 는 `reasoning_effort` (low/medium/high). 현재 favor 코드에 **부재** → 모든 agent 가 default (medium) 으로 균질하게 호출됨.

#### 현재 코드

`util/llm_client.py:75-88` 의 `call_llm()` signature 에 `reasoning_effort` 인자 없음.

#### 변경안

```python
def call_llm(
    model: str,
    system_prompt: str,
    user_prompt: str,
    ...
    temperature: float = 0.3,
    reasoning_effort: Optional[Literal["low", "medium", "high"]] = None,  # 신규
    ...
):
    ...
    kwargs = {
        "model": model,
        "messages": messages,
        "temperature": temperature,
    }
    if reasoning_effort is not None:
        kwargs["reasoning_effort"] = reasoning_effort
    resp = client.chat.completions.create(**kwargs)
```

각 agent 의 호출에 reasoning_effort 매핑 추가:
- `hypothesis_agent.py:185`: `reasoning_effort="high"` (창의성)
- `validation_agent.py:754`: `reasoning_effort="low"` (속도/엄격)
- `factor_coder.py / costeer.py`: `reasoning_effort="low"`
- `formula_agent.py / observation_agent.py`: `reasoning_effort="medium"` (default)

#### 영향 추정

- OpenAI hidden-reasoning 모델 (gpt-5.4-mini, o-series) 에서만 effect. 다른 백본 (gpt-4o, Qwen, Llama 등) 은 unknown param 무시 → 영향 없음.
- 비용 측면: reasoning_effort=high 는 hidden reasoning token 더 많이 생성 → 비용 ↑. validation 의 low 는 비용 ↓.
- mean shift lever 가 될 수 있는 첫 lever (temperature 는 variance).

#### 영향 등급

- 가벼운 패치지만 hidden-reasoning 모델 호출 시 처음으로 agent 별 차별화 가능. Phase B (백본 비교) 의 fairness 보강.
